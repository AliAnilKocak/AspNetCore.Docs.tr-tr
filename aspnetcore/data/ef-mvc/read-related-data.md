### <a name="performance-considerations"></a><span data-ttu-id="5df61-136">Performans değerlendirmeleri</span><span class="sxs-lookup"><span data-stu-id="5df61-136">Performance considerations</span></span>

<span data-ttu-id="5df61-142">Diğer taraftan, bazı senaryolarda ayrı sorgulara daha verimli.</span><span class="sxs-lookup"><span data-stu-id="5df61-142">On the other hand, in some scenarios separate queries is more efficient.</span></span> <span data-ttu-id="5df61-143">Bir sorgu tüm ilgili veriler istekli yüklenmesini hangi SQL Server verimli bir şekilde işleyemiyor oluşturulması, bir çok karmaşık birleştirme neden olabilir.</span><span class="sxs-lookup"><span data-stu-id="5df61-143">Eager loading of all related data in one query might cause a very complex join to be generated, which SQL Server can't process efficiently.</span></span> <span data-ttu-id="5df61-144">Veya yalnızca bir alt kümesini işleme varlık kümesi için bir varlığın gezinme özelliklerinin erişmeniz gerekiyorsa, ayrı sorgulara Önden her şeyi istekli yüklenmesini ihtiyacınız daha fazla veri almak çünkü daha iyi gerçekleştirebilir.</span><span class="sxs-lookup"><span data-stu-id="5df61-144">Or if you need to access an entity's navigation properties only for a subset of a set of the entities you're processing, separate queries might perform better because eager loading of everything up front would retrieve more data than you need.</span></span> <span data-ttu-id="5df61-145">Performans önemliyse, performansı en iyi seçim yapmak için her iki yönde test en iyisidir.</span><span class="sxs-lookup"><span data-stu-id="5df61-145">If performance is critical, it's best to test performance both ways in order to make the best choice.</span></span>
